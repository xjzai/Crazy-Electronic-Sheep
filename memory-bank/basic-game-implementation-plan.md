# 《电子羊会发疯》基础游戏实施计划

## 1. 文档目的

本计划只服务于“基础游戏闭环”开发，不覆盖完整 MVP，不覆盖长期运营版。  
本计划面向 AI 开发者，要求按顺序执行，每一步完成后必须先验证，再进入下一步。

## 2. 本轮范围

### 包含

- 单地图基础游玩场景
- 单货币：摸鱼能量
- 自动产出
- 购买基础羊
- 羊在地图内自由走动显示
- 拖拽与两两合成
- 首次解锁新羊
- 基础图鉴记录
- 点击查岗收益
- 本地存档与读档
- 基础离线收益
- 调试面板与基础日志

### 不包含

- 第二地图及跨地图解锁
- 科技系统
- 广告系统
- 免费随机羊
- 排行榜
- 好友、社交、分享
- 后端服务
- 热更新
- 分包优化
- 活动系统

## 3. 执行约束

- 每一步只解决一个明确问题，不要顺手扩展未来功能。
- 每一步必须留下可观察结果，不能只完成“内部准备”。
- 每一步必须附带验证测试，并亲自执行验证。
- 如某一步失败，先修复当前问题，不允许跳步。
- 所有数值先使用最小可调版本，避免一开始做复杂平衡。
- 所有配置优先外置，不要把基础数值硬编码到多个位置。

## 4. 完成定义

当本计划全部完成时，基础游戏应满足以下结果：

- 玩家进入游戏后能看到单地图主界面
- 资源会自动增长
- 玩家可以购买羊并让它在地图中自由走动
- 玩家可以拖拽羊并完成合成
- 玩家可以通过点击羊获得额外收益
- 图鉴能记录首次解锁的羊
- 关闭并重新打开后能恢复进度
- 离线一段时间后能正确补发基础收益
- 全程无明显报错，无阻断性死状态

## 5. 分步指令

### 步骤 1：创建基础项目并锁定版本

目标：
建立一个可启动、可预览、可继续开发的 Cocos Creator 基础项目。

指令：
明确本项目使用的 Cocos Creator 版本、脚本语言和平台目标。
创建空白项目并确认默认启动流程正常。
记录项目版本信息，确保团队后续使用同一编辑器版本。

验证测试：
打开项目并进入编辑器主界面。
运行一次本地预览。
确认能进入空白场景，且控制台没有阻断性报错。

完成标准：
项目能稳定打开，预览正常，版本信息明确。

### 步骤 2：建立最小工程目录约定

目标：
为后续开发固定资源、脚本、配置、场景和测试的摆放位置。

指令：
建立统一目录结构，并明确命名规则。
把“运行时逻辑”“配置数据”“UI 资源”“场景资源”“测试资源”分开。
记录每类目录的用途，避免后续混放。

验证测试：
检查目录结构是否完整。
随机抽查 3 类资源，确认能明确判断其归属位置。
确认没有“临时目录”“未命名目录”“重复职责目录”。

完成标准：
目录职责清晰，后续开发者看到工程后能立刻定位文件放置位置。

### 步骤 3：搭建启动场景与主场景骨架

目标：
让项目具备明确的启动入口和基础游戏主场景。

指令：
建立启动场景和基础游戏主场景。
启动场景只负责初始化和跳转，不承担业务逻辑。
主场景保留基础区域占位：顶部资源区、中部地图区、底部操作区。

验证测试：
运行项目后确认先进入启动场景，再进入主场景。
检查主场景是否按竖屏布局显示三段区域。
确认场景切换过程无白屏卡死、无重复进入。

完成标准：
场景入口固定，主场景骨架可用于后续系统接入。

### 步骤 4：确定基础分辨率与屏幕适配规则

目标：
让基础界面在常见竖屏比例下不出现严重错位。

指令：
锁定基础设计分辨率。
确定安全区域策略、顶部资源栏最小高度、底部操作栏最小高度。
先保证核心信息可见，不做复杂动效适配。

验证测试：
在至少 3 种竖屏比例下预览。
确认顶部资源、地图区域、底部按钮没有互相遮挡。
确认地图漫游区域完整可见，且不会被顶部或底部区域压缩到无法使用。

完成标准：
基础 UI 在主流竖屏比例下均能正常显示。

### 步骤 5：建立基础配置文件并定义首批字段

目标：
把基础游戏依赖的关键数值和羊配置集中到配置层。

指令：
建立基础配置文件，先只覆盖首轮需要的最小字段。
至少包含：基础羊配置、秒产、购买价、点击倍率、最大活动羊数量、地图漫游边界、出生最小间距、行走速度范围、待机时长范围、初始资源、离线结算上限。
字段名称保持稳定，避免之后频繁改名。

验证测试：
逐项检查配置字段是否足够支撑单地图闭环。
确认同一数值没有同时出现在多个来源。
移除一项关键字段进行验证，确认系统能明确暴露缺失问题而不是静默出错。

完成标准：
基础配置可支撑后续逻辑开发，字段边界清楚。

### 步骤 6：建立配置加载与校验流程

目标：
保证游戏启动时能安全读取配置，并在配置错误时尽早失败。

指令：
实现配置加载顺序和校验规则。
校验内容至少包括：羊编号连续性、购买价有效性、秒产有效性、重复编号检查、最大活动羊数量大于零、地图漫游边界合法、行走速度范围合法、待机时长范围合法、出生最小间距大于零。
启动流程中必须先校验配置，再进入主游戏。

验证测试：
用一份正确配置启动游戏，确认加载通过。
故意制造一份错误配置，确认启动阶段能给出明确失败信息。
恢复正确配置后再次启动，确认流程恢复正常。

完成标准：
配置问题能在启动期暴露，不会拖到游戏运行中才随机报错。

### 步骤 7：建立全局游戏状态模型

目标：
统一管理当前资源、已拥有羊、已解锁羊、图鉴进度和存档状态。

指令：
定义基础游戏状态的最小数据边界。
明确哪些数据属于“运行时临时状态”，哪些数据属于“必须存档状态”。
避免让 UI 直接持有业务真值。

验证测试：
列出至少 10 个核心状态项，逐一判断是否应进入全局状态。
检查同一状态是否只存在一个可信来源。
确认关闭某个 UI 后，游戏主状态不会丢失。

完成标准：
状态边界明确，后续系统都能围绕同一份真值工作。

### 步骤 8：实现资源自动增长主循环

目标：
让游戏在没有任何玩家输入时也能持续产出资源。

指令：
实现基础时间推进机制。
资源增长应来自当前已放置羊的总秒产。
资源变化要有统一更新入口，不允许多个地方各自加钱。

验证测试：
启动后不做任何操作，观察资源是否持续增长。
放置 1 只羊和放置多只羊时，资源增长速率应明显不同。
暂停时间推进后，资源应停止增长；恢复后应继续增长。

完成标准：
自动产出稳定、可预测、可调试。

### 步骤 9：实现自由漫游地图模型

目标：
建立单地图的自由漫游区域规则，为购买、走动、拖拽、合成提供基础。

指令：
定义地图中的可行走边界、安全边距和最大活动羊数量。
建立基础位置约束规则，确保羊不会走出边界。
建立出生点与落点的最小间距规则，避免羊长期重叠在同一点。

验证测试：
初始化地图时，漫游区域应为空且边界明确。
放入一只羊后，该羊应出现在合法区域内。
连续放入多只羊后，位置应基本分散，不应全部堆叠在同一点。

完成标准：
地图边界和活动密度规则稳定可用。

### 步骤 10：实现基础羊实例生成规则

目标：
让系统能根据配置生成羊实例，并绑定等级与产出信息。

指令：
建立基础羊实例模型。
实例必须能表达：羊编号、等级、当前位置、目标位置、朝向、移动状态、秒产、是否已解锁、是否可点击。
不要在这一步加入科技、广告加成、多地图归属等未来字段。

验证测试：
创建多个不同等级羊实例，检查其基础属性是否正确。
检查同编号羊实例在地图中能独立存在并保持各自位置。
销毁一个实例后，其他实例状态不应受影响。

完成标准：
羊实例创建、存在、移除行为稳定。

### 步骤 11：实现随机走动状态机

目标：
让每只羊在地图中看起来像“自己在乱逛”，而不是静止摆件。

指令：
建立最小走动状态机，至少包含待机、选点、移动、到达四个阶段。
每只羊在待机结束后，应随机选择一个新的合法目标点并向其移动。
目标点应尽量与周围羊保持基础间距，不要求复杂避障。
羊向左移动时应水平反转贴图，向右移动时恢复原向。

验证测试：
单只羊在地图中应持续出现待机与走动切换。
羊移动到左侧目标点时，贴图方向应反转；移动到右侧目标点时应恢复。
连续运行一段时间后，羊不应频繁走出边界或卡死在边界线上。

完成标准：
基础走动逻辑稳定，贴图转向规则正确。

### 步骤 12：实现购买系统

目标：
让玩家能消耗资源购买基础羊，并生成到地图中。

指令：
实现购买入口、资源扣除、购买失败处理和生成逻辑。
首轮只允许购买当前基础羊，不引入招聘机范围规则。
资源不足和地图达到最大活动羊数量时，必须给出明确反馈。
购买成功后，应为新羊分配一个合法出生位置并立即进入基础走动循环。

验证测试：
资源足够时，购买后应扣除正确资源并新增一只羊。
资源不足时，购买应失败且资源不变。
地图达到活动羊上限时，购买应失败且不生成隐藏实例。

完成标准：
购买逻辑对“成功、余额不足、达到活动羊上限”三种情况处理正确。

### 步骤 13：实现羊入场显示与基础表现

目标：
让购买出来的羊能在地图上被稳定看见、识别和观察到走动。

指令：
为羊实例绑定基础视觉节点。
确保玩家能一眼区分不同等级羊，且能明显看出它们在地图中移动。
当前阶段只做最小必要表现，不投入复杂动画，但必须保证左右转向的贴图反转可见。

验证测试：
连续购买多只羊，确认每只羊都出现在合法位置并开始走动。
观察至少一只羊向左和向右移动时，确认贴图反转正确。
切换界面状态后返回，显示不应错位、消失或停止刷新。

完成标准：
地图显示和实例状态一一对应，玩家能正常识别当前局面。

### 步骤 14：实现拖拽交互与拖拽期间暂停走动

目标：
让玩家能稳定抓住一只正在走动的羊并进行拖拽操作。

指令：
实现基础拖拽流程：选中、拖动、释放。
拖拽开始后，被选中的羊必须停止自主走动。
为降低操作难度，拖拽开始后其余羊也应暂时停止随机走动，直到拖拽结束。
释放到合法地图区域但未触发合成时，应将羊放在释放点附近的合法位置，再恢复走动。

验证测试：
拖起一只正在走动的羊，确认它立即停止自主移动。
拖拽过程中观察其他羊，确认它们已暂时停止走动。
释放后若未合成，羊应落在合法位置并恢复随机走动。

完成标准：
拖拽交互稳定，且不会因目标持续移动造成明显操作挫败。

### 步骤 15：实现两两合成规则

目标：
建立基础游戏最核心的成长动作。

指令：
实现“同等级两只羊合成为下一等级一只羊”的规则。
合成后应正确移除两只原羊，生成一只新羊。
不允许不同等级强行合成。

验证测试：
拖拽两个同等级羊重叠后释放，确认生成下一等级羊。
拖拽不同等级羊到一起后释放，确认不合成且状态恢复正确。
连续执行多次合成，确认不会出现等级跳错、丢羊、重复扣除。

完成标准：
合成结果稳定，状态与显示完全一致。

### 步骤 16：实现首次解锁记录

目标：
让“第一次合出更高等级羊”成为可追踪的进度。

指令：
在首次获得某个羊编号时，记录为已解锁。
已解锁记录必须独立于当前地图上是否还存在该羊。
后续重复获得同编号羊时，不应重复触发首次解锁逻辑。

验证测试：
首次合出新羊时，确认解锁记录新增。
卖出、移除或继续合成后，解锁记录仍保留。
再次获得相同新羊时，不应重复记为首次。

完成标准：
解锁进度可信，可作为图鉴基础数据源。

### 步骤 17：实现基础图鉴界面

目标：
给玩家一个可见的收集反馈出口。

指令：
建立最小图鉴界面。
未解锁项和已解锁项要能明确区分。
当前阶段只展示基础信息：编号、名称、是否解锁。

验证测试：
初始进入图鉴时，确认大部分项为未解锁。
解锁新羊后重新打开图鉴，确认状态变化正确。
关闭并重新打开图鉴多次，确认显示一致。

完成标准：
图鉴能稳定反映真实解锁状态。

### 步骤 18：实现查岗点击收益

目标：
补齐基础游戏中的主动交互收益点。

指令：
允许玩家点击地图中的羊获取额外资源。
点击收益必须基于该羊当前秒产。
当前阶段只做单羊冷却，不做全局能量池。

验证测试：
点击一只羊时，资源应立即增加。
短时间重复点击同一只羊，冷却中不应反复给收益。
点击不同等级羊时，收益应体现明显差异。

完成标准：
点击收益可感知，但不会压过自动产出主循环。

### 步骤 19：实现顶部资源区与关键 HUD

目标：
让玩家持续看见当前核心进度信息。

指令：
在主界面显示至少这些信息：当前资源、当前总秒产、当前最高解锁羊、当前图鉴进度。
资源变化要及时刷新。
数字显示规则保持统一，不做过早的大数字复杂缩写。

验证测试：
购买、点击、合成后，相关 HUD 应立即更新。
长时间挂机后，总资源和秒产显示应保持正确。
进入图鉴再返回主界面，HUD 不应错乱。

完成标准：
玩家能随时理解自己当前处于什么进度。

### 步骤 20：实现本地存档

目标：
保证玩家关闭游戏后不会丢失基础进度。

指令：
为基础游戏建立本地存档读写规则。
至少存储：资源、羊实例列表、羊当前位置、已解锁羊、最高解锁羊、上次退出时间。
存档写入点要明确，不要每帧写入。

验证测试：
进行一轮购买、合成、解锁后退出游戏。
重新进入游戏，确认主要进度恢复正确。
存档为空时应走新档流程，存档损坏时应有安全回退方案。

完成标准：
读档和新档路径都清晰，基础进度可恢复。

### 步骤 21：实现基础离线收益

目标：
让放置游戏的核心承诺在基础版本中成立。

指令：
根据上次退出时间和当前总秒产结算离线收益。
设置基础离线结算上限，避免异常时间导致收益失真。
离线收益领取流程要明确，不能静默吞掉。

验证测试：
退出一段短时间后重进，确认获得合理离线收益。
退出超出上限时间后重进，确认收益受上限约束。
没有可结算离线时间时，不应错误弹出奖励。

完成标准：
离线收益可解释、可验证、不过量。

### 步骤 22：实现失败反馈与异常保护

目标：
降低玩家和开发者面对错误状态时的混乱成本。

指令：
为资源不足、地图达到活动羊上限、非法拖放、错误配置、坏档恢复设置明确反馈。
任何失败都应“无副作用失败”，不能半成功半失败。
关键异常要进入统一日志出口。

验证测试：
依次制造 5 种常见失败场景，确认都能稳定反馈。
失败后再次执行正常操作，系统应仍可恢复工作。
检查日志中是否能定位失败原因。

完成标准：
失败路径清楚，异常不扩散。

### 步骤 23：加入调试面板

目标：
提升调试效率，避免每次验证都靠手工长流程。

指令：
加入基础调试入口。
至少支持：加资源、清存档、快速生成低级羊、快速生成高级羊、显示当前状态摘要。
调试能力只能在开发模式可见。

验证测试：
使用调试面板快速构造合成、图鉴、离线、达到活动羊上限场景。
确认调试操作结果与正常逻辑一致。
切到非开发模式时，调试入口不可见。

完成标准：
主要验证场景都能在 1 分钟内构造出来。

### 步骤 24：执行基础功能回归测试

目标：
确认各系统组合后仍然稳定。

指令：
按“新档开始 -> 购买 -> 随机走动 -> 拖拽 -> 合成 -> 解锁 -> 点击 -> 存档 -> 重进 -> 离线收益”的顺序执行完整回归。
记录每一步预期结果和实际结果。
任何异常都必须先修复，再重新跑完整回归。

验证测试：
完整跑通至少 3 次。
其中 1 次必须包含达到活动羊上限场景。
其中 1 次必须包含坏档或空档启动验证。

完成标准：
三轮完整回归全部通过。

### 步骤 25：执行数值与节奏初调

目标：
让基础版本具备“可玩”而不是只有“能运行”。

指令：
调整初始资源、基础购买价、基础秒产、点击倍率、离线上限，使前几分钟的循环清晰成立。
先追求节奏顺畅，不追求完整商业化数值。
记录本轮定稿参数与调整理由。

验证测试：
从新档开始体验 10 到 15 分钟。
确认玩家能稳定进入“购买 -> 合成 -> 解锁新羊”的循环。
确认点击收益有价值，但不会压过放置主收益。

完成标准：
基础版本已具备可试玩性。

### 步骤 26：输出基础游戏验收清单

目标：
为后续继续加功能提供一个可信的阶段交付基线。

指令：
整理一份基础游戏验收清单。
明确“已完成”“未完成”“故意后置”的内容。
把当前已知问题单独列出，不允许混在已完成项里。

验证测试：
用验收清单重新对照实际游戏。
确认每项都能在当前版本里被观察和验证。
让另一位开发者按清单走一遍，确认没有歧义。

完成标准：
基础游戏阶段边界清晰，可以作为下一阶段的稳定起点。

## 6. 建议执行顺序检查

在真正动手前，AI 开发者应再次确认以下顺序不被打乱：

1. 先启动与配置，再做主循环。
2. 先状态与漫游区，再做购买、走动与合成。
3. 先核心玩法闭环，再做图鉴与存档。
4. 先稳定性与调试能力，再做数值调优。

## 7. 阶段产出物

- 可运行的基础游戏工程
- 基础配置文件
- 基础图鉴数据
- 本地存档结构说明
- 调试面板
- 基础回归测试记录
- 基础游戏验收清单
