# Project AGENTS

本文件用于约束本仓库内任何 AI 代码生成行为。  
如果本文件与临时口头要求冲突，以本文件和 `memory-bank` 中的核心文档为准，除非用户明确要求修改规则本身。

## Always

以下规则为始终应用，任何代码生成前都必须执行：

1. 写任何代码前必须完整阅读 `memory-bank/architecture.md`（包含当前完整架构与当前持久化数据结构；若未来引入数据库，必须先在该文件中补全完整数据库结构）。
   说明：
   该文件是当前系统架构、数据流和持久化结构的唯一权威文档。当前阶段虽然没有后端数据库，但该文件已经明确了当前本地存档结构；未来如果引入数据库，必须先在该文件中补全数据库结构，再开始实现。

2. 写任何代码前必须完整阅读 `memory-bank/game-design-document.md`。
   说明：
   该文件是玩法、边界、核心循环和当前设计约束的唯一权威文档。

3. 写任何基础游戏相关代码前必须完整阅读 `memory-bank/basic-game-implementation-plan.md`。
   说明：
   该文件定义了基础游戏闭环的正确实现顺序、验证测试和完成标准。

4. 写任何涉及技术选型、工程结构、平台接入、测试、存储、监控的代码前，必须完整阅读 `memory-bank/tech-stack.md`。

5. 每完成一个重大功能或里程碑后，必须更新 `memory-bank/architecture.md`。

6. 每完成一个重大功能或里程碑后，必须更新 `memory-bank/progress.md`。

7. 在核心文档之间发现冲突时，不允许直接忽略。必须先：
   - 明确冲突点
   - 选择权威来源
   - 必要时先更新文档
   - 再开始编码

8. 始终采用模块化、多文件、单一职责设计。
   强制要求：
   - 优先拆分为多个小文件
   - 一个文件只承担一个清晰职责
   - 一个模块只处理一个主要领域问题
   - UI、状态、领域逻辑、平台适配、存储、配置加载必须分离

9. 明确禁止单体巨文件和 Monolith。
   禁止行为：
   - 创建“God Object”“God Manager”“万能 Controller”“万能 Service”
   - 把 UI、状态管理、网络、存档、配置解析写进同一个文件
   - 把多个系统硬塞进一个超长文件只因为“方便改”
   - 在已有大文件上继续无边界累积逻辑而不拆分
   - 当单文件已经明显跨越多个职责时，继续往同一文件堆逻辑

10. 任何新增代码都必须先考虑“应该放进哪个现有模块”，如果现有模块职责不合适，就新建小模块，而不是继续堆到单一文件中。

11. 写任何代码时必须写清晰、详细、可维护的注释。
   强制要求：
   - 变量定义处要写清楚这个变量是干什么的
   - 如果变量有单位、取值范围、生命周期或特殊约束，也要写清楚
   - 每个函数或方法都要写清楚它是干什么的
   - 函数注释至少要说明：用途、关键输入、关键输出、主要副作用
   - 对不直观的状态字段、流程控制变量、缓存变量、标记位变量，必须解释其存在意义
   - 对复杂分支、边界条件、时序逻辑、平台差异逻辑，必须补充解释性注释
   - 禁止只写“赋值给变量”“调用函数”“更新状态”这类没有信息量的废话注释
   - 注释必须帮助后来者理解“为什么这样写”和“这个东西负责什么”，而不只是重复代码字面意思

## 架构约束

1. 以 `memory-bank/architecture.md` 为当前实现的模块边界基线。
2. 当前基础游戏阶段采用自由漫游地图，不允许重新引入格子模型，除非先更新设计文档和架构文档。
3. 当前基础游戏阶段没有后端数据库，不允许臆造服务端表结构并直接编码。
4. 如果未来要引入后端、数据库、排行榜、云存档或账号系统，必须先更新 `memory-bank/architecture.md`，再开始实现。

## 模块拆分规则

推荐的代码拆分方向：

- 场景入口文件
- 启动协调器
- 视图层文件
- 状态容器
- 领域服务
- 平台适配器
- 配置加载器
- 存档仓库
- 调试工具
- 类型定义文件

拆分要求：

- 不同系统的职责边界必须明显
- 不同模块之间通过明确接口协作
- 纯逻辑优先从视图层剥离
- 平台相关逻辑不得散落到 UI 和领域逻辑内部
- 如果一个文件超过约 300 行且已经承担多个职责，必须优先拆分，而不是继续追加
- 如果一个文件超过约 500 行，默认视为高风险 monolith 候选，除非有非常明确且可辩护的单一职责理由

## 技术栈规则

1. 默认按 `memory-bank/tech-stack.md` 执行。
2. 客户端逻辑优先遵循：
   - `Cocos Creator`
   - `TypeScript`
   - 配置驱动
   - 本地存档
   - 平台 Adapter 分层

3. 不要在没有充分理由的情况下引入新的重框架。
4. 不要把数值、平台开关、走动参数分散硬编码到多个文件。
5. 与测试、脚本、工程化相关的变更要保持与 `pnpm`、`ESLint`、`Prettier`、`Vitest` 方案一致。

## 注释规则

1. 注释不是可选优化，而是默认要求。
2. 任何核心变量、状态结构、函数、服务入口都应有说明性注释。
3. 对领域逻辑的注释，优先解释业务语义，而不是只解释语法动作。
4. 对走动、合成、离线收益、存档、平台适配等关键系统，必须让注释足够详细到新接手的模型或开发者可以直接继续维护。
5. 如果代码已经明显不直观，但没有补注释，视为未完成实现。

## 状态与数据规则

1. 必须维持单一可信状态来源。
2. UI 不得持有业务真值。
3. 可从配置恢复的静态数据不应写入存档。
4. 当前阶段的持久化真值以本地存档结构为准。
5. 状态字段新增、删除或重命名时，必须同步检查：
   - `memory-bank/architecture.md`
   - 存档兼容性
   - 调试入口
   - 相关测试

## 网络与平台规则

1. 没有平台 Adapter 时，不要把平台 API 直接写进业务逻辑。
2. 没有后端需求时，不要提前写伪网络层制造复杂度。
3. 一旦引入网络请求：
   - 网络调用必须与 UI 分离
   - 错误处理必须集中
   - 数据结构必须可验证

## 测试与验证规则

1. 任何重大功能都必须配套验证步骤。
2. 如果 `memory-bank/basic-game-implementation-plan.md` 已定义验证测试，必须执行对应验证。
3. 涉及以下内容时，优先补充或更新测试：
   - 数值公式
   - 合成逻辑
   - 随机走动逻辑
   - 点击收益
   - 存档读写
   - 离线收益

## 文档同步规则

以下情况发生时，必须先或立即更新文档：

- 模块边界变化
- 数据结构变化
- 存档结构变化
- 核心玩法变化
- 地图交互模型变化
- 技术栈变化

最少需要同步的文档：

- `memory-bank/architecture.md`
- `memory-bank/progress.md`
- 必要时同步 `memory-bank/game-design-document.md`
- 必要时同步 `memory-bank/basic-game-implementation-plan.md`

## 审查清单

生成或修改代码前，先自检：

1. 是否完整阅读了 `memory-bank/architecture.md`？
2. 是否完整阅读了 `memory-bank/game-design-document.md`？
3. 当前任务若属于基础游戏，是否完整阅读了 `memory-bank/basic-game-implementation-plan.md`？
4. 这次改动是否会制造单体巨文件？
5. 能否把本次改动拆成多个职责清晰的小文件？
6. 是否把 UI、状态、逻辑、存储、平台适配混写在一起了？
7. 是否已经为关键变量、关键函数、复杂流程补了清晰注释？
8. 本次改动完成后，是否需要更新 `memory-bank/architecture.md` 和 `memory-bank/progress.md`？
